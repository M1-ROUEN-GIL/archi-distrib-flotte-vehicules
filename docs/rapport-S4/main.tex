\section{Objectifs des nouveaux services}

\begin{infobox}
\textbf{Objectif :} Implémenter les microservices de \textbf{Maintenance} (suivi technique) et \textbf{Driver} (gestion des conducteurs et permis) pour assurer la conformité et la disponibilité de la flotte.
\end{infobox}

Cette itération se concentre sur deux piliers essentiels :
\begin{itemize}
    \item \textbf{Service Maintenance} : Responsable de la santé technique via la planification d'interventions et la détection de retards.
    \item \textbf{Service Driver} : Gestion administrative des conducteurs, incluant le suivi critique de la validité des permis de conduire.
\end{itemize}

\section{Architecture de Persistance}

Le système respecte strictement le pattern \textbf{Database per Service}. Chaque microservice métier dispose de son propre serveur PostgreSQL (StatefulSet) et de sa propre base de données, garantissant une isolation totale des données.

\subsection{Isolation des instances}
\begin{itemize}
    \item \textbf{Véhicules} : Instance \texttt{postgres-service} (base \texttt{flotte\_db}).
    \item \textbf{Maintenance} : Instance \texttt{postgres-maintenance-service} (base \texttt{maintenance\_db}).
    \item \textbf{Driver} : Instance \texttt{postgres-driver-service} (base \texttt{driver\_db}).
\end{itemize}

\subsection{Initialisation via Kubernetes}
L'initialisation des schémas est gérée par des \textbf{ConfigMaps} dédiés injectés dans les conteneurs au démarrage :
\begin{itemize}
    \item \texttt{flotte-db-init} : Schémas pour les véhicules et la localisation.
    \item \texttt{maintenance-db-init} : Schéma \texttt{03-maintenance.sql}.
    \item \texttt{driver-db-init} : Schéma \texttt{02-driver.sql} et données initiales (\textit{seeds}).
\end{itemize}

\section{Microservice Maintenance}

Le service Maintenance assure la traçabilité complète des interventions sur les véhicules.

\subsection{Entité MaintenanceRecord}
L'entité centrale \texttt{MaintenanceRecord} gère les cycles de vie des interventions via plusieurs énumérations (\texttt{Type}, \texttt{Status}, \texttt{Priority}).

\begin{lstlisting}[language=Java, caption=Entité MaintenanceRecord]
@Entity
@Table(name = "maintenance_records")
public class MaintenanceRecord {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    
    private UUID vehicleId; // Lien logique vers le Service Vehicules
    
    @Enumerated(EnumType.STRING)
    private MaintenanceStatus status = MaintenanceStatus.SCHEDULED;
    
    private LocalDate scheduledDate;
    private Integer nextServiceKm; // Seuil pour les alertes preventives
}
\end{lstlisting}

\subsection{Logique Métier et Alertes}

Le service implémente des mécanismes proactifs pour assurer le suivi de la maintenance.

\subsubsection{Gestion des retards (Overdue)}
Une tâche planifiée (\texttt{@Scheduled}) s'exécute quotidiennement à 01h00 pour identifier les maintenances prévues dont la date est dépassée.

\begin{lstlisting}[language=Java, caption=Tâche de vérification des retards]
@Scheduled(cron = "0 0 1 * * ?")
@Transactional
public void checkOverdueMaintenance() {
    List<MaintenanceRecord> scheduledRecords = repository.findByStatus(MaintenanceStatus.SCHEDULED);
    LocalDate today = LocalDate.now();
    for (MaintenanceRecord record : scheduledRecords) {
        if (record.getScheduledDate().isBefore(today)) {
            record.setStatus(MaintenanceStatus.OVERDUE);
            repository.save(record);
            eventProducer.publishAlert(AlertEvent.overdue(
                record.getVehicleId(), "Maintenance en retard"));
        }
    }
}
\end{lstlisting}

\section{Microservice Driver}

Le service Driver gère le cycle de vie des conducteurs et la conformité de leurs permis de conduire.

\subsection{Entité Driver}
Le service utilise une relation \textbf{Un-à-Plusieurs} entre l'entité \texttt{Driver} et \texttt{DriverLicense}. Il implémente également le \textit{Soft Delete} via Hibernate.

\begin{lstlisting}[language=Java, caption=Entité Driver]
@Entity
@Table(name = "drivers")
@SQLDelete(sql = "UPDATE drivers SET deleted_at = CURRENT_TIMESTAMP WHERE id = ?")
@SQLRestriction("deleted_at IS NULL")
public class Driver {
    @Id
    @GeneratedValue(strategy = GenerationType.UUID)
    private UUID id;
    
    private String firstName;
    private String lastName;
    
    @Enumerated(EnumType.STRING)
    private DriverStatus status = DriverStatus.active;

    @OneToMany(mappedBy = "driver", cascade = CascadeType.ALL)
    private List<DriverLicense> licenses = new ArrayList<>();
}
\end{lstlisting}

\subsection{Logique Métier et Conformité}

Le service Driver assure la validité administrative des missions.

\subsubsection{Surveillance des Expirations (LicenseScheduler)}
Une tâche planifiée vérifie chaque nuit la validité des permis enregistrés pour alerter les gestionnaires avant l'expiration effective.

\begin{lstlisting}[language=Java, caption=Vérification proactive des permis]
@Scheduled(cron = "0 0 0 * * *")
@Transactional(readOnly = true)
public void checkLicenseExpirations() {
    // Detection des permis expirant dans 30 jours
    LocalDate in30Days = LocalDate.now().plusDays(30);
    licenseRepository.findByExpiryDate(in30Days).forEach(license -> {
        eventProducer.sendLicenseExpiringEvent(license);
    });

    // Detection des permis expires aujourd'hui
    LocalDate today = LocalDate.now();
    licenseRepository.findByExpiryDate(today).forEach(license -> {
        eventProducer.sendLicenseExpiredEvent(license);
    });
}
\end{lstlisting}

\subsection{Synchronisation par Événements (Kafka)}
Le bus d'événements assure la cohérence entre les services sans couplage direct.

\subsubsection{Topic : flotte.maintenance.events}
Le service \textbf{Véhicules} consomme ces messages pour ajuster l'état opérationnel :
\begin{itemize}
    \item \texttt{MAINTENANCE\_STARTED} $\rightarrow$ Statut véhicule : \texttt{in\_maintenance}.
    \item \texttt{MAINTENANCE\_COMPLETED/CANCELLED} $\rightarrow$ Statut véhicule : \texttt{available}.

\end{itemize}

\subsubsection{Topic : flotte.driver.events}
Ce topic notifie les créations et mises à jour de conducteurs, ainsi que les alertes d'expiration de permis détectées par le \texttt{LicenseScheduler}.

\subsection{Corrections Techniques Apportées}
L'implémentation a nécessité la résolution de plusieurs défis techniques majeurs rencontrés lors des tests d'intégration :
\begin{itemize}
    \item \textbf{Routage et redirection (405 Method Not Allowed)} : Identification et correction d'une erreur 405 lors des requêtes \texttt{PUT}. Le problème provenait d'un conflit de port entre l'Ingress et le service Kubernetes, ainsi que d'une gestion ambiguë des slashes finaux dans les contrôleurs Spring Boot.
    \item \textbf{Désérialisation Kafka (ClassCastException)} : Résolution d'une erreur où les événements reçus étaient interprétés comme des \texttt{LinkedHashMap} au lieu des types Records attendus. La solution a consisté à configurer explicitement le \texttt{spring.json.type.mapping} dans les propriétés Kafka.
    \item \textbf{Cohérence des Schémas GraphQL} : Correction du crash du Gateway au démarrage dû à l'absence des types de pagination (\texttt{DriverPage}, \texttt{AlertPage}) dans les fichiers de définition du schéma.
    \item \textbf{Contraintes de base de données} : Gestion des erreurs 500 liées à la contrainte \texttt{chk\_completed\_date} (vérifiant que la fin d'une maintenance ne précède pas sa date prévue).
\end{itemize}

\subsection{Exposition via le Gateway GraphQL}
L'ensemble des services (Véhicule, Driver, Maintenance) est fédéré par une passerelle GraphQL implémentée avec Apollo Server.

\subsubsection{Implémentation du Gateway (Apollo Server)}
Une application Node.js dédiée centralise et expose le schéma GraphQL. Les requêtes sont résolues via un \texttt{Context} instanciant des clients REST (Axios) pour chaque microservice :
\begin{itemize}
    \item \textbf{VehicleClient} : requêtes vers \texttt{http://vehicle-service:8080}
    \item \textbf{DriverClient} : requêtes vers \texttt{http://driver-service:8080}
    \item \textbf{MaintenanceClient} : requêtes vers \texttt{http://maintenance-service:8082}
\end{itemize}

\subsubsection{Agrégation multi-services}
L'API Gateway permet d'agréger les données de façon transparente pour le client :
\begin{itemize}
    \item \textbf{Maintenance $\leftrightarrow$ Conducteur} : Résolution du technicien.
    \item \textbf{Véhicule $\leftrightarrow$ Maintenance} : Affichage de l'historique technique depuis la fiche véhicule.
    \item \textbf{Véhicule $\leftrightarrow$ Conducteur} : Détail de l'assignation courante d'un véhicule vers le profil du conducteur (et ses permis).
\end{itemize}

\subsubsection{Conteneurisation et Ingress}
Le Gateway est buildé dans une image Docker \texttt{graphql-gateway:latest} et déployé sur le cluster Kubernetes (port 4000). La configuration de l'\textbf{Ingress} route désormais le trafic de \texttt{/graphql} vers cette passerelle.

\section{Architecture Événementielle (Saga Pattern)}

Le système implémente le pattern Saga par \textbf{Chorégraphie}, assurant la cohérence éventuelle entre les services distribués.

\subsection{Transactions Compensatoires}
Pour garantir l'intégrité des données, un mécanisme de compensation a été mis en \oe{}uvre sur le flux de maintenance :
\begin{enumerate}
    \item \textbf{T1 (Action)} : Le service Maintenance crée ou démarre une intervention (\texttt{MAINTENANCE\_STARTED}).
    \item \textbf{T2 (Réaction)} : Le service Véhicule tente de passer le véhicule en \texttt{in\_maintenance}.
    \item \textbf{C1 (Compensation)} : En cas d'échec de T2, le service Véhicule publie un événement \texttt{MAINTENANCE\_REJECTED}. Le service Maintenance consomme cet événement et \textbf{annule} automatiquement la fiche d'intervention (\texttt{cancelRecord}) pour restaurer l'état initial.

\end{enumerate}

\subsection{Fiabilité des échanges (Outbox Pattern)}
Une réflexion a été menée sur la fiabilité de la "double écriture" (Base de données + Kafka). Bien que l'\textbf{Outbox Pattern} soit le standard industriel pour garantir l'atomicité parfaite entre la persistance et la publication d'événements, il n'a pas été retenu pour cette itération.

Compte tenu de la taille du projet et de la complexité opérationnelle qu'impliquerait la mise en place d'un outil comme \textit{Debezium} ou d'un worker de relayage, nous avons privilégié le \textbf{Direct Publishing} au sein de transactions Spring \texttt{@Transactional}. Cette approche, combinée aux transactions compensatoires du pattern Saga, offre un excellent compromis entre robustesse et maintenabilité pour notre cas d'usage.

\subsection{Flux de données}
L'architecture repose sur deux modes de communication complémentaires pour assurer la fluidité et la cohérence des données.

\subsubsection{Flux Synchrones (Lecture et Agrégation)}
Les requêtes de consultation transitent par le \textbf{Gateway GraphQL}. Ce dernier joue le rôle d'orchestrateur :
\begin{itemize}
    \item Il reçoit une requête unique (ex: fiche véhicule avec historique et conducteur).
    \item Il appelle en parallèle les services REST concernés via leurs APIs respectives.
    \item Il assemble les réponses JSON pour renvoyer un objet structuré au client.
\end{itemize}

\subsubsection{Flux Asynchrones (Événements et Saga)}
Les changements d'état critiques utilisent le bus de messages \textbf{Kafka} pour garantir le découplage :
\begin{itemize}
    \item \textbf{Cycle de vie Maintenance} : \texttt{Maintenance $\rightarrow$ Kafka $\rightarrow$ Véhicule} (Mise à jour automatique de la disponibilité).
    \item \textbf{Mise à jour Kilométrique} : \texttt{Véhicule $\rightarrow$ Kafka $\rightarrow$ Maintenance} (Calcul des prochaines échéances et alertes).
    \item \textbf{Assignation Chauffeur} : \texttt{Véhicule $\rightarrow$ Kafka $\rightarrow$ Driver} (Passage du chauffeur en statut \texttt{on\_tour}).
\end{itemize}

\section{Tests automatisés et couverture de code}

\begin{infobox}
\textbf{Objectif :} Garantir la robustesse via une couverture minimale de 80\%.
\end{infobox}

\subsection{Stratégie et Résultats}
L'assurance qualité repose sur une pyramide de tests automatisés couvrant l'intégralité des trois services métier (\textbf{Véhicules}, \textbf{Maintenance}, \textbf{Driver}).

\begin{itemize}
    \item \textbf{Tests Unitaires} : Utilisation de \textit{JUnit 5} et \textit{Mockito} pour valider 100\% des méthodes de la couche \texttt{Service}. Cela inclut la logique critique des changements d'états, les calculs kilométriques et les tâches planifiées (\textit{Schedulers}).
    \item \textbf{Tests d'Intégration} : Utilisation de \textit{MockMvc} et d'une base de données \textit{H2} en mémoire pour valider le routage des APIs, la désérialisation JSON et le respect des règles de sécurité RBAC (Keycloak).
    \item \textbf{Couverture JaCoCo} : La mesure automatisée via le plugin \textit{JaCoCo} confirme une couverture supérieure à 80\% sur le périmètre métier de chaque microservice, garantissant ainsi la non-régression lors des évolutions futures.
\end{itemize}

\section{Observabilité et Supervision}

L'ensemble de l'architecture distribuée est couplé à une stack d'observabilité moderne permettant une supervision en temps réel de la santé des services.

\subsection{Collecte centralisée (OpenTelemetry)}
Chaque microservice métier ainsi que le Gateway GraphQL sont configurés pour exporter leurs données télémétriques vers un \textbf{OpenTelemetry Collector} centralisé. Cette approche standardisée garantit l'indépendance vis-à-vis des outils de stockage final.

\subsection{Trois piliers de l'observabilité}
\begin{itemize}
    \item \textbf{Logs (Grafana Loki)} : Les logs de tous les conteneurs sont agrégés et indexés. L'utilisation de l'agent OpenTelemetry Java permet d'inclure automatiquement les contextes de traces dans les logs applicatifs.
    \item \textbf{Métriques (Prometheus)} : Le système expose des indicateurs de performance (CPU, mémoire, temps de réponse des APIs) collectés par Prometheus.
    \item \textbf{Traces distribuées (Jaeger)} : Les requêtes transitant par le Gateway GraphQL vers les services REST sont tracées de bout en bout, permettant d'identifier rapidement les goulots d'étranglement ou les échecs dans les échanges asynchrones.
\end{itemize}

\subsection{Visualisation (Grafana)}
Un tableau de bord Grafana unique permet de croiser ces trois sources de données, offrant aux administrateurs une vision consolidée du système (disponible sur le port 3001 en local).
